2. Встановлення zod

zod — це TypeScript-first бібліотека валідації даних. Вона дозволяє описати схему даних і автоматично перевіряти вхідні дані на відповідність цій схемі. Zod став стандартом у Next.js екосистемі завдяки простоті та нульовим залежностям.

Завдання: Встановити zod

Terminal
npm install zod

Переконайтеся, що zod з’явився в package.json:

package.json (фрагмент)
"dependencies": {
  ...
  "zod": "^3.x.x"
}
Чому zod? Легка бібліотека без залежностей, стандарт у Next.js екосистемі. Працює і на сервері, і на клієнті. Автоматично виводить TypeScript-типи зі схем.
Checkpoint

zod є в dependencies у package.json.

3. Створення схем валідації

Zod-схема описує структуру даних: які поля обов’язкові, які типи допустимі, обмеження на довжину, формат тощо. Якщо вхідні дані не відповідають схемі — zod поверне список помилок з людино-зрозумілими повідомленнями.

Завдання: Створити схему валідації для напоїв

Створіть файл lib/validations/drink.js:

lib/validations/drink.js
import { z } from "zod";

export const createDrinkSchema = z.object({
  name: z
    .string({ required_error: "Назва обов'язкова" })
    .min(1, "Назва обов'язкова")
    .max(100, "Максимум 100 символів")
    .trim(),
  description: z
    .string()
    .max(500, "Максимум 500 символів")
    .trim()
    .optional()
    .default(""),
  price: z
    .number({
      required_error: "Ціна обов'язкова",
      invalid_type_error: "Ціна має бути числом",
    })
    .min(0, "Ціна не може бути від'ємною"),
  emoji: z.string().max(10).optional().default("☕"),
  category: z.enum(["Кава", "Чай", "Їжа", "Інше"], {
    errorMap: () => ({
      message: "Категорія має бути: Кава, Чай, Їжа або Інше",
    }),
  }),
  available: z.boolean().optional().default(true),
});

// Для PUT — всі поля необов'язкові
export const updateDrinkSchema = createDrinkSchema.partial();

Метод .partial() робить усі поля необов’язковими — це ідеально для PUT-запитів, де клієнт може надіслати лише ті поля, які хоче змінити.

Завдання: Створити схеми валідації для користувачів

Створіть файл lib/validations/user.js:

lib/validations/user.js
import { z } from "zod";

export const registerSchema = z.object({
  name: z
    .string({ required_error: "Ім'я обов'язкове" })
    .min(2, "Мінімум 2 символи")
    .max(50, "Максимум 50 символів")
    .trim(),
  email: z
    .string({ required_error: "Email обов'язковий" })
    .email("Некоректний формат email")
    .toLowerCase()
    .trim(),
  password: z
    .string({ required_error: "Пароль обов'язковий" })
    .min(6, "Мінімум 6 символів")
    .max(100, "Максимум 100 символів"),
});

export const updateRoleSchema = z.object({
  role: z.enum(["user", "admin"], {
    errorMap: () => ({
      message: "Роль має бути 'user' або 'admin'",
    }),
  }),
});
Zod vs Mongoose: Zod-схема дзеркалює Mongoose-модель, але з іншою метою. Mongoose валідує при збереженні в БД. Zod валідує на вході в API — раніше і з кращими повідомленнями про помилки. Це два різних рівні захисту.
Checkpoint

Обидва файли створено: lib/validations/drink.js та lib/validations/user.js. Синтаксичних помилок немає.

4. Валідація API напоїв

Зараз наші API endpoints приймають будь-які дані і передають їх напряму в Mongoose. Додамо валідацію через zod — перед тим як дані потраплять до бази.

Завдання: Додати валідацію до POST /api/drinks

Відкрийте app/api/drinks/route.js та оновіть функцію POST:

app/api/drinks/route.js
import { NextResponse } from "next/server";
import dbConnect from "@/lib/db";
import Drink from "@/models/Drink";
import { authorize } from "@/lib/authorize";
import { createDrinkSchema } from "@/lib/validations/drink";

export async function GET() {
  await dbConnect();
  const drinks = await Drink.find({}).sort({ createdAt: -1 });
  return NextResponse.json(drinks);
}

export async function POST(request) {
  const { session, error } = await authorize("admin");
  if (error) return error;

  await dbConnect();

  try {
    const data = await request.json();

    // Валідація через zod
    const result = createDrinkSchema.safeParse(data);
    if (!result.success) {
      const messages = result.error.errors.map((e) => e.message);
      return NextResponse.json({ errors: messages }, { status: 400 });
    }

    const drink = await Drink.create(result.data);

    return NextResponse.json(drink, { status: 201 });
  } catch (error) {
    if (
      error.message === "Unexpected end of JSON input" ||
      error instanceof SyntaxError
    ) {
      return NextResponse.json(
        { error: "Невалідний JSON у тілі запиту" },
        { status: 400 }
      );
    }
    return NextResponse.json(
      { error: "Помилка сервера" },
      { status: 500 }
    );
  }
}
safeParse() vs parse(): safeParse() повертає об’єкт { success, data, error } і ніколи не кидає виняток. parse() кидає ZodError — його потрібно ловити в try/catch. Для API зручніший safeParse().

Завдання: Додати валідацію до PUT /api/drinks/[id]

Відкрийте app/api/drinks/[id]/route.js та оновіть функцію PUT:

app/api/drinks/[id]/route.js (фрагмент PUT)
import { updateDrinkSchema } from "@/lib/validations/drink";

// В PUT handler:
export async function PUT(request, { params }) {
  const { session, error } = await authorize("admin");
  if (error) return error;

  await dbConnect();

  try {
    const { id } = await params;
    const data = await request.json();

    // Валідація через zod
    const result = updateDrinkSchema.safeParse(data);
    if (!result.success) {
      const messages = result.error.errors.map((e) => e.message);
      return NextResponse.json({ errors: messages }, { status: 400 });
    }

    const drink = await Drink.findByIdAndUpdate(id, result.data, {
      new: true,
      runValidators: true,
    });

    if (!drink) {
      return NextResponse.json(
        { error: "Напій не знайдено" },
        { status: 404 }
      );
    }

    return NextResponse.json(drink);
  } catch (error) {
    return NextResponse.json(
      { error: "Помилка сервера" },
      { status: 500 }
    );
  }
}
Важливо: Ми передаємо result.data (а не data) до Mongoose. Це означає, що в БД потраплять тільки валідовані поля — зайві поля з запиту будуть автоматично відкинуті zod’ом.
Checkpoint

Протестуйте за допомогою Postman, Thunder Client або curl:

  • POST /api/drinks з { "name": "", "price": -5 } → 400 з масивом помилок
  • POST /api/drinks з { "name": {"$gt": ""}, "price": 50, "category": "Кава" } → 400 (name не є рядком)
  • POST /api/drinks з валідними даними → 201

5. Валідація API реєстрації

Зараз API реєстрації має ручну валідацію (if (!name || !email)). Замінимо її на zod — це дасть кращі повідомлення про помилки та автоматичну нормалізацію даних (trim, toLowerCase).

Завдання: Замінити ручну валідацію на zod в API реєстрації

Відкрийте app/api/auth/register/route.js та оновіть:

app/api/auth/register/route.js
import { NextResponse } from "next/server";
import bcrypt from "bcryptjs";
import dbConnect from "@/lib/db";
import User from "@/models/User";
import { registerSchema } from "@/lib/validations/user";
import { stripHtml } from "@/lib/sanitize";

export async function POST(request) {
  try {
    await dbConnect();

    const data = await request.json();

    // Валідація через zod (замість ручної)
    const result = registerSchema.safeParse(data);
    if (!result.success) {
      const messages = result.error.errors.map((e) => e.message);
      return NextResponse.json(
        { error: messages.join(", ") },
        { status: 400 }
      );
    }

    const { email, password } = result.data;
    const name = stripHtml(result.data.name);

    // Перевірка унікальності email
    const existingUser = await User.findOne({ email });
    if (existingUser) {
      return NextResponse.json(
        { error: "Користувач з таким email вже існує" },
        { status: 409 }
      );
    }

    const hashedPassword = await bcrypt.hash(password, 10);

    const user = await User.create({
      name,
      email,
      password: hashedPassword,
    });

    return NextResponse.json(
      {
        message: "Користувача успішно створено",
        user: {
          id: user._id,
          name: user.name,
          email: user.email,
          role: user.role,
        },
      },
      { status: 201 }
    );
  } catch (error) {
    if (error.code === 11000) {
      return NextResponse.json(
        { error: "Користувач з таким email вже існує" },
        { status: 409 }
      );
    }
    return NextResponse.json(
      { error: "Помилка сервера" },
      { status: 500 }
    );
  }
}
Що робить zod автоматично: перевіряє формат email, обрізає пробіли (trim()), конвертує email у нижній регістр (toLowerCase()). Ручні перевірки if (!name || !email) більше не потрібні.
Увага: імпорт stripHtml

Функція stripHtml буде створена на кроці 7 (Санітизація). Поки що цей імпорт викличе помилку — це нормально. Ви можете тимчасово закоментувати рядок імпорту та використання stripHtml, або спочатку виконати крок 7.

Checkpoint

Протестуйте реєстрацію:

  • POST /api/auth/register з порожнім email → 400 «Email обов’язковий»
  • POST /api/auth/register з паролем «123» → 400 «Мінімум 6 символів»
  • POST /api/auth/register з валідними даними → 201

6. Валідація API користувачів

API зміни ролі (PUT /api/users/[id]) приймає поле role. Без валідації хтось може надіслати { "role": "superadmin" } або взагалі будь-який рядок. Додамо zod-валідацію.

Завдання: Додати валідацію до PUT /api/users/[id]

Відкрийте app/api/users/[id]/route.js та оновіть функцію PUT:

app/api/users/[id]/route.js (фрагмент PUT)
import { updateRoleSchema } from "@/lib/validations/user";

// В PUT handler:
export async function PUT(request, { params }) {
  const { session, error } = await authorize("admin");
  if (error) return error;

  await dbConnect();

  try {
    const { id } = await params;
    const data = await request.json();

    // Валідація через zod
    const result = updateRoleSchema.safeParse(data);
    if (!result.success) {
      const messages = result.error.errors.map((e) => e.message);
      return NextResponse.json(
        { error: messages.join(", ") },
        { status: 400 }
      );
    }

    const { role } = result.data;

    const user = await User.findByIdAndUpdate(
      id,
      { role },
      { new: true }
    ).select("-password");

    if (!user) {
      return NextResponse.json(
        { error: "Користувача не знайдено" },
        { status: 404 }
      );
    }

    return NextResponse.json(user);
  } catch (error) {
    return NextResponse.json(
      { error: "Помилка сервера" },
      { status: 500 }
    );
  }
}
Перевага zod enum: Якщо хтось надішле { "role": "superadmin" }, zod автоматично поверне помилку — enum дозволяє тільки "user" або "admin". Без валідації такий рядок потрапив би в БД.
Checkpoint

Протестуйте:

  • PUT /api/users/[id] з { "role": "superadmin" } → 400 «Роль має бути ‘user’ або ‘admin’»
  • PUT /api/users/[id] з { "role": "admin" } → 200 (роль змінено)

7. Санітизація даних

Валідація перевіряє структуру та типи даних. Санітизація — це очищення вмісту: видалення HTML-тегів, потенційно небезпечного коду тощо. Це додатковий рівень захисту (defence in depth).

Завдання: Створити хелпер санітизації

Створіть файл lib/sanitize.js:

lib/sanitize.js
/**
 * Видаляє HTML-теги з рядка
 */
export function stripHtml(str) {
  if (typeof str !== "string") return str;
  return str.replace(/<[^>]*>/g, "").trim();
}

/**
 * Санітизує всі рядкові поля об'єкта
 */
export function sanitizeObject(obj) {
  const sanitized = {};
  for (const [key, value] of Object.entries(obj)) {
    sanitized[key] = typeof value === "string" ? stripHtml(value) : value;
  }
  return sanitized;
}

Завдання: Інтегрувати санітизацію в API напоїв

Оновіть app/api/drinks/route.js — додайте санітизацію після валідації:

app/api/drinks/route.js (фрагмент POST)
import { sanitizeObject } from "@/lib/sanitize";

// Після валідації через zod:
const result = createDrinkSchema.safeParse(data);
if (!result.success) {
  const messages = result.error.errors.map((e) => e.message);
  return NextResponse.json({ errors: messages }, { status: 400 });
}

// Санітизація
const sanitized = sanitizeObject(result.data);
const drink = await Drink.create(sanitized);

Зробіть аналогічно для PUT в app/api/drinks/[id]/route.js:

app/api/drinks/[id]/route.js (фрагмент PUT)
import { sanitizeObject } from "@/lib/sanitize";

// Після валідації:
const sanitized = sanitizeObject(result.data);
const drink = await Drink.findByIdAndUpdate(id, sanitized, {
  new: true,
  runValidators: true,
});
Чому санітизація після валідації? Zod перевіряє структуру та типи. Санітизація очищує вміст. Порядок важливий: спочатку переконуємося, що дані правильного типу, потім чистимо вміст рядків.
Важливо розуміти

Санітизація не замінює React JSX escaping! React вже автоматично екранує вивід у JSX. Санітизація — додатковий рівень (defence in depth), щоб у БД не потрапляв HTML-код. Якщо ви використовуєте dangerouslySetInnerHTML — тоді санітизація на сервері критично важлива.

Checkpoint

Протестуйте:

  • POST /api/drinks з name "<script>alert('xss')</script>Лате" → у БД збережено як "alert('xss')Лате" (теги видалено)
  • Звичайні назви без HTML-тегів зберігаються без змін

8. Безпекові заголовки

HTTP-заголовки безпеки — це інструкції для браузера, як обробляти контент з вашого сайту. Вони захищають від типових атак: clickjacking, MIME-sniffing тощо. Next.js дозволяє налаштувати заголовки через next.config.mjs.

Завдання: Додати безпекові заголовки

Відкрийте (або створіть) файл next.config.mjs у корені проєкту та оновіть:

next.config.mjs
/** @type {import('next').NextConfig} */
const nextConfig = {
  async headers() {
    return [
      {
        source: "/(.*)",
        headers: [
          {
            key: "X-Content-Type-Options",
            value: "nosniff",
          },
          {
            key: "X-Frame-Options",
            value: "DENY",
          },
          {
            key: "X-XSS-Protection",
            value: "0",
          },
          {
            key: "Referrer-Policy",
            value: "strict-origin-when-cross-origin",
          },
          {
            key: "Permissions-Policy",
            value: "camera=(), microphone=(), geolocation=()",
          },
        ],
      },
    ];
  },
};

export default nextConfig;

Що робить кожний заголовок:

  • X-Content-Type-Options: nosniff — запобігає MIME-type sniffing (браузер не буде «вгадувати» тип контенту)
  • X-Frame-Options: DENY — захист від clickjacking (забороняє завантаження сайту в iframe)
  • X-XSS-Protection: 0 — вимикає застарілий XSS-фільтр браузера (сучасний підхід — Content Security Policy)
  • Referrer-Policy — контролює, яка інформація передається в HTTP-заголовку Referrer при переході на інший сайт
  • Permissions-Policy — обмежує доступ сторінки до камери, мікрофона та геолокації

Завдання: Перевірити заголовки в браузері

Після зміни next.config.mjs потрібно перезапустити сервер:

Terminal
# Зупиніть сервер (Ctrl+C) та запустіть знову
npm run dev

Перевірте заголовки:

  1. Відкрийте сайт у браузері
  2. Відкрийте DevTools (F12)
  3. Перейдіть на вкладку Network
  4. Оновіть сторінку
  5. Натисніть на будь-який запит (наприклад, перший документ)
  6. У вкладці Headers перегляньте Response Headers
  7. Знайдіть всі 5 заголовків
Нагадування: Після зміни next.config.mjs завжди потрібно перезапустити сервер (npm run dev). Зміни в цьому файлі не підхоплюються автоматично.

Бонусні завдання

Ці завдання не є обов’язковими і не впливають на наступні практичні роботи.

Бонус 1: Rate Limiting

Додайте простий rate limiter для /api/auth/register — обмеження кількості запитів з одного IP за хвилину. Використайте Map для зберігання лічильників.

Бонус 2: CORS конфігурація

Додайте CORS headers через middleware для API-маршрутів. Дозвольте запити тільки з вашого домену.

Бонус 3: Content Security Policy

Додайте базовий CSP header до next.config.mjs. Дозвольте тільки ресурси з вашого домену (self).
